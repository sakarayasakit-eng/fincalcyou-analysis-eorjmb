## Executive Summary

fin·calc is a solo-built, AI-assisted, ad-revenue-motivated static finance-calculator site: one 417 KB `index.html` containing 14 interactive calculators for 34 countries (all inline CSS + JS, no framework, no build step), plus 298 SEO landing pages in `/pages/` and 12 DSCR pages at root. The math engine is small and correct (one reducing-balance EMI formula plus a dozen derived formulas, backed by an in-page self-test suite), and the SEO plumbing (sitemap generator, IndexNow CI, hreflang clusters, JSON-LD) is unusually complete for a project of this size.

The core architectural finding: **there is no source of truth anywhere in the repo.** The 298 landing pages are *materialized output of AI generation sessions whose template no longer exists*. The country dataset (rates, bank names, product notes) exists in at least four independent copies — `index.html`'s `COUNTRY_PROFILES`/`BANKS` JS objects, each landing page's prose, each landing page's FAQ JSON-LD, and `fix_seo.py`'s `COUNTRY`/`BANK` Python dicts — and the mini-calculator engine is pasted inline into 267 pages in three divergent variants. The two Python scripts at root are not page generators for `/pages/`; `generate_pages.py` only creates the 11 DSCR pages + sitemap, and `fix_seo.py` is a one-off regex *patcher* that exists precisely because there is no template to regenerate from. Every sitewide change is therefore a bespoke migration script or a 300-file hand edit.

The fix is not a framework. It is a committed data registry (countries/banks/products/pages as JSON) plus a deterministic local generator that emits the same static HTML currently in git — preserving CLAUDE.md's "no build step at deploy time" guardrail while collapsing ~4.9 MB of hand-frozen HTML into ~50 KB of data + ~30 KB of templates.

### 1.2 Application architecture

- **No framework, no build step, no modules — confirmed.** `package.json` has no build script (`"test"` stub only), `type: commonjs`, dependencies: `@anthropic-ai/sdk` (runtime) and `netlify-cli` (dev). **The Anthropic SDK is used by nothing**: `grep -rl "anthropic"` across all `.js/.html/.py` excluding node_modules returns zero hits. It is a leftover from AI-assisted dev sessions and can be removed.
- **`index.html` layout:** lines 1–210 head/meta/schema; 211–1304 one `<style>` block (~1,100 lines); 1306–2867 body (welcome modal, header, country bar, tab grid, 14 panel `<div class="card panel">` blocks, explore section, footer, modals); 2868–7483 one `<script>` block (~4,600 lines) — data objects, then per-calculator functions, then init/UX code.
- **Rendering flow:** static HTML → browser parses everything (all 14 panels in DOM, hidden via `.panel{display:none}` index.html:366) → inline script runs at parse end: builds currency picker, applies country profile, `applyUrlParams()` (:6198), `detectByIp()` (:6223), wires ~70 sliders with `input` listeners, `recomputeAll()` (:6184) recalculates all 16 calculators on any global change.
- **Routing:** file-system routing only. Deep links via query params (`?cur=PKR&tab=emi&amount=...` — index.html:6196) and one Netlify redirect (`_redirects`: `/pages/sip-calculator-india → /pages/sip-calculator-india-20-years 301`).
- **State management:** plain global variables — `currentCur` (index.html:4178), `rates`/`ratesDate` (:4123–4124), plus `localStorage` keys `fincalc_cur`, `fincalc_rates` (1-hour FX cache), `fincalc_theme`, saved scenarios (:6632–6639, capped at 5), calc-count, welcome-seen, jargon toggle (grep results at :4139–:7107). No stores, no reactivity — every function reads DOM inputs directly by `getElementById`.
- **Hooks / component hierarchy / hydration / code splitting / dynamic imports / build output: N/A** — these concepts do not exist here because there is no framework and no bundler. The closest analogue to "components" is (a) the repeated `.field` slider markup pattern in index.html and (b) the `customElements.define('fc-…')` inline web components on landing pages (e.g. pages/car-loan-calculator-japan.html:117) — which are *copy-pasted per page*, not shared.
- **External runtime dependencies (all client-side fetch):** Google Fonts (render-blocking `<link>` index.html:209), `api.exchangerate.host` for FX (:4149, with `FALLBACK_RATES` hardcoded at :4125), `get.geojs.io` for IP country detection (:6227), and commented-out AdSense loader (:27–29).
- **Serverless:** exactly one function, `netlify/functions/lead_capture.js` — parses `{email, calc_type, url}`, forwards to Resend API with `RESEND_API_KEY` env var, hardcoded recipient `sakarayasakit@gmail.com` and sandbox sender `onboarding@resend.dev` (lead_capture.js:20–21). Only the 12 DSCR pages call it (verified: `grep -rl lead_capture|script.js` matches only `dscr-*.html`).
- **CI:** `.github/workflows/indexnow.yml` on every push to main: runs `scripts/update-sitemap-lastmod.js` (stamps `<lastmod>` from `git log -1` per file), commits sitemap if changed, then `scripts/submit-indexnow.js` POSTs all 315 sitemap URLs to `api.indexnow.org` with the committed key.
- **Environment variables:** `RESEND_API_KEY` (Netlify) is the only one used. `.env` is empty; `.env.txt` contains a placeholder `ANTHROPIC_API_KEY=YOUR_CLAUDE_API_KEY` (not a real secret); both are gitignored (though `.env.txt` sat on disk — commit `deb83bc2 "Stop tracking env files"` shows they were previously tracked).
- **Dependency graph between subsystems:** `index.html` ⇄ `/pages/` link to each other (pages deep-link into tabs via `../index.html?cur=JPY&tab=car&...` — car-loan-calculator-japan.html:167; index.html's Explore section links to hubs at :2816–2833). Pages share only `landing.css`; ALL page JS is inline per page. DSCR pages share `script.js` + `pages/landing.css`. No circular *code* dependency exists because almost nothing is shared — the coupling is **data duplication**, not imports (Phase 8).

## Phase 2 – Understand the Business Logic

Grounded in PROJECT_REVIEW_v19.md, STRATEGIC_ADVISORY.md, README.md, CLAUDE.md and page copy; labeled where inferred.

- **Business model (documented):** AdSense display revenue. PROJECT_REVIEW_v19.md states the origin explicitly: *"The original goal was always AdSense / AdMob revenue"*, and records the pivot from a Flutter app to a web calculator because *"finance web calculators earn $4–$10+ CPM"*. `index.html` contains four dormant ad slots (`#adSlotTop` :1437, `#adSlotEmiMid` :1629, `#adSlotContent` :2763, `#adSlotFooter` :2836) hidden until a real `<ins class="adsbygoogle">` exists (`.ad-slot:not(:has(ins.adsbygoogle)){display:none}` :503), plus `ads.txt`. Ads are deliberately suppressed on the Islamic tab (*"trust signal for Muslim users"* :500).
- **Secondary monetization experiment (recent):** the DSCR cluster is a **lead-generation** play for US real-estate investors — the only part of the site with an email gate (`dscr-calculator.html:84`, `lead_capture.js`). This directly contradicts the sitewide "No signup. No email capture. No tracking" positioning (index.html trust strip :1419–1425, llms.txt) — see Phase 10/11.
- **Audience segmentation (documented in STRATEGIC_ADVISORY.md competitor analysis):** Tier-3 markets (India, Pakistan, Bangladesh — "EMI" terminology, SIP culture, Islamic finance) are the core wedge where NerdWallet/Bankrate are weak; the Gulf (Zakat, Murabaha, gratuity pages), and long-tail global English queries ("mortgage calculator canada"). README.md: *"This is the calculator's most visited tab in tier-3 markets"* (SIP) — **HYPOTHESIS as stated; no analytics in repo to verify** (site claims no tracking, so presumably no GA — would need Netlify Analytics/Search Console to confirm).
- **Why each calculator exists:** the 14 tabs map to the highest-volume finance query classes per market: EMI/mortgage (universal head term), car, education (South Asia moratorium loans), SIP (India), FD (South Asia/Gulf savings culture), Zakat/Murabaha/Halal (Muslim markets, 1.8B addressable), retirement, rent-vs-buy, affordability, compare, debt payoff, health check (retention/engagement feature per PROJECT_REVIEW "product with retention" pivot), currency converter (remittance corridors — the 21 converter pages are all remittance pairs like AED→PKR, USD→INR).
- **Why the homepage differs from landing pages:** homepage = product (interactive, stateful, everything in one file for a returning user); landing pages = SEO surface area (one keyword each, thin embed calculator, prose, FAQ schema, deep link into the product). This is a deliberate hub-and-spoke documented in CLAUDE.md ("8 calculator hub pages … choose your country links block (hub-and-spoke internal linking)").
- **What every homepage section is for:** welcome modal (first-visit onboarding → goal + country), country bar (credibility: named central bank + policy rate + bank picker), trust strip (differentiators as claims: "34 countries / 340 banks / no tracking"), tab grid (navigation), per-tab interpretation cards & example buttons (financial-literacy audience, v20 "friendliness features" per CSS comment :611), self-test modal ("✓ Check calculator math" :2804 — a trust device: recomputes worked examples live), Explore section (:2812–2834, funnels PageRank and users to /pages/), FAQ content block (:2765–2803, AEO answer fodder).

### 3.D Clustering: which calculators are the same engine

- **One engine, many skins:** EMI, Car, Education, Compare, Afford, Rent-vs-Buy, and the amortization/prepay tools are all the single reducing-balance formula (or its inverse) with different wrappers. ## Phase 5 – Deep Analysis of the Guide Calculators (the "guides directory" + landing pages)

Cards' descriptions are hand-written summaries duplicating each target page's meta description.
- **Landing-page anatomy** (from full reads of car-loan-calculator-japan/kenya, home-loan-calculator, what-is-emi, india-financial-calculators, usd-to-inr-converter, hdfc-fd-calculator spot checks): head meta + canonical + og/twitter + FAQPage JSON-LD + BreadcrumbList + Organization/WebSite `@graph` (added sitewide — car-loan-calculator-japan.html:65); hreflang cluster (33 alternates + x-default, in 173 pages per grep); H1 + last-updated `<time>` stamp (2026-07-03, patched sitewide per commit `9248fb97`); inline embed CSS + embed calculator + inline engine script; "Local rates & providers" prose; "Worked example" with baked numbers; visible FAQ mirroring the JSON-LD; deep-link CTA to `../index.html?cur=…&tab=…`; "Related calculators" list (generated by fix_seo.py `make_related_links` :509); footer.

## Phase 6 – Reverse Engineer the Country System

The country system lives entirely in index.html's script and is the app's real crown jewel — and its most duplicated asset.

- **Registries (all in index.html):**
  1. `currencies[]` — 50 currencies with symbol/flag/name (:2872–2924).
  2. `COUNTRY_PROFILES` — 34 entries keyed by currency code (:2932–3701): `country, centralBank, policyRate, loanRate, carRate, carTenureMax, savingsRate, investmentReturn` + 8 localized product name/desc/note strings each (~20 lines/country). `ASOF = 'July 2026'` (:2931).
  3. `BANKS` — 34 country arrays × ~10 banks with `[homeMin,homeMax]`/`[carMin,carMax]` ranges + notes (:3709–4120) — the "340 banks" claim checks out.
  4. `SMART_DEFAULTS` — 37 entries of realistic per-country slider defaults (:4212–4258).
  5. Detection maps: `TZ_CUR` timezone→currency + `EUR_TZ` + `REGION_CUR` locale→currency (:4326–4356) and `GEO_REGION_CUR` ISO-country→currency (:6218–6222).
  6. Behavior lists: `SOUTH_ASIA_CODES` (EMI wording :4183), `ISLAMIC_CODES` (19 codes, tab visibility :4419).
- **Detection & storage cascade (verified order):** `?cur=` URL param (`applyUrlParams` :6198) → `localStorage['fincalc_cur']` (:4364) → timezone/locale heuristic (`detectCurrencyCode` :4325) → async IP lookup via GeoJS that only runs when neither saved choice nor param exists (`detectByIp` :6223–6244). Choice persists on manual pick (:4313). CLAUDE.md's cache-buster warning exists because of this persistence.
- **Application:** `applyCountryProfile()` (:4387) rewrites tab titles, H2s, descriptions, product notes, hero labels (EMI↔Monthly Payment), sets car tenure max, populates the bank picker, sets rate chips; `applySmartDefaults()` fires synthetic `input` events so all listeners recompute (:4260–4283); `recomputeAll()` (:6184) runs all 16 calculators.
- **Currency vs country conflation:** the system is keyed by **currency code**, not country. EUR collapses 11+ eurozone countries into one "Eurozone" profile (fine), but it also means Oman/Jordan/Lebanon/Nepal/Afghanistan/Iran currencies exist in `currencies[]` with **no profile** — the picker labels profiled ones "LOCAL RATES" (:4300). Tax, salary and pension rules are **absent by design** (disclaimed at index.html:2776: "Tax implications … are not applied").
- **Scalability judgment:** adding country #35 requires touching ≥6 JS registries in index.html + generating ~8 landing pages + editing hreflang blocks in ~33×N_products existing files + adding cards to pages/index.html + fix_seo.py's `COUNTRY` dict. The runtime design scales fine (O(1) lookup); the **authoring** design scales O(products × countries) with manual hreflang O(n²) — this is where it fails (Phase 13).
- **Where it succeeds:** terminology localization (EMI/payment, SIP/DCA/Reksa Dana), Islamic market adaptation, realistic per-market defaults (IDR 800,000,000 vs USD 350,000 — :4231 vs :4220), bank benchmarking. This is materially better than the "giant" competitors' non-US coverage, exactly as STRATEGIC_ADVISORY.md claims.

## Phase 7 – Reverse Engineer the Interlinking System

Every link in the system is **hardcoded HTML**; two generators exist that *write* hardcoded links:

- **Landing page → related pages:** the "Related calculators" `<ul>` on 280 pages (grep `<h2>Related`) is generated by `fix_seo.py::make_related_links` (:509–636) — semantic rules per category (home→car/fd/sip/rentbuy/retire of same country + affordability + India flagship; converter→home/sip of both corridor countries; glossary→topical calculator via `term_target` map :622–631). Link targets are existence-checked against the directory (`add_link` :499–506) — good hygiene. But the output is frozen: rename/delete a page and 280 pages hold stale anchors until the script is re-run and re-tuned.
- **Hubs → spokes:** product hubs carry 32-country link blocks (home-loan-calculator.html:108–141); country hubs carry per-country tool grids (india-financial-calculators.html:32–43); the guides directory links everything (297 cards); root index Explore links 8 hubs + DSCR + 5 country hubs (:2816–2833) and the footer links `/pages/` sitewide.
- **Spokes → hub/app:** breadcrumb JSON-LD (Home→Guides→page), header nav, and the deep-link CTA into the app with prefilled state.
- **hreflang mesh:** 173 pages carry reciprocal 33-alternate clusters with `x-default` → **the root app** `https://fincalcyou.netlify.app/` (car-loan-calculator-japan.html:44) — not the product hub. Defensible (the app is the generic answer) but it concentrates x-default equity on a page whose title targets different terms.
- **DSCR cluster:** self-contained mesh (city ↔ locations hub ↔ general calculator, generate_pages.py:96,109–115) + one Explore link from the homepage (:2824). It is otherwise isolated from /pages/.
- **Risks of the hardcoded approach:** (1) O(n²) hreflang maintenance; (2) fix_seo.py's `classify()` (:190–303) is a regex tower over *filename conventions* — new slug patterns silently fall to `"unknown"` and get skipped; (3) the guides directory and sitemap and related-links are three independent registries of "what pages exist" that can drift (the sitemap corruption fixed in commit `7be8127d` is recorded evidence of exactly this drift).

## Phase 13 – Scalability Audit (299 → 500 → 1,000 → 5,000 pages)

Mechanism-specific, based on the maintenance plane found in Phases 5–8:

- **→ 500 pages (adds ~2 product clusters or 1 country):** every new *country* costs: 6 registry edits in index.html + ~8 new pages + **hreflang edits to ~200 existing files** (33-line block per page in each affected cluster — this is the O(n²) killer, already the largest mechanical burden) + guides-directory cards + fix_seo.py `COUNTRY` entry. Feasible only via AI sessions; error rate compounds (evidence it already happens: commit `8a1f5ec7` fixing missed self-hreflang on Indonesian bank pages, commit `7be8127d` fixing a corrupted sitemap).
- **→ 1,000 pages:** pages/index.html doubles to ~230 KB with 700+ DOM cards — the directory becomes the heaviest page on the site and its substring search degrades UX; fix_seo.py's `classify()` regex tower needs new branches per naming pattern or silently skips pages; sitemap generation and IndexNow still fine (they scale linearly and are automated); *auditability* is gone — no human can diff 1,000 hand-frozen files, so a data-drift bug (wrong rate on 40 pages) becomes undetectable.
- **→ 5,000 pages:** the repo is ~80 MB of HTML (plus the already-committed node_modules); git operations and Netlify deploys still work (Netlify handles far larger sites — hosting is NOT the constraint), but every sitewide change (rate year bump "2026→2027", footer link, embed engine fix) is a 5,000-file rewrite that only a script can do — and the only scripts that exist are single-purpose regex patchers. IndexNow submits all URLs on every push (submit-indexnow.js posts the full list — fine, API allows 10k/batch). The **real ceiling is editorial**: 5,000 thin pages on a subdomain with no backlinks will not index (CLAUDE.md's own SEO-reality note), so scaling page count before authority is negative-value work.
- **index.html growth:** each new country adds ~35 lines (profile + banks + defaults); at 100 countries the file is ~460 KB — still functional but the inline-data share crosses 50%. The single-file design degrades gracefully in runtime terms and terribly in maintenance terms.
- **What does NOT break:** Netlify hosting/CDN, the sitemap/IndexNow automation, the runtime country system, page self-containment.

## Phase 14 – The Real Engineering Problem

**The user's implicit framing ("too much duplication, needs shared components") is directionally right but names the symptom, not the disease.** The disease, stated precisely:

> **This codebase has outputs but no sources.** The 298 landing pages are compiled artifacts of AI chat sessions — a mail-merge whose template and data file were thrown away after each run. The repo therefore contains ~4.9 MB of frozen HTML whose true information content is ~50 KB (34 country profiles, 340 bank rows, ~15 product definitions, ~300 page specs) plus 3 small templates.

Evidence chain: (1) no generator for /pages/ exists — `generate_pages.py` covers only DSCR (read in full, markets list :7–18); (2) `fix_seo.py` is a 742-line *reverse-engineering tool* that re-derives page semantics from filenames (`classify()` :190–303) in order to patch titles — you only write that script when the forward pipeline is gone; (3) the same Japan car-loan sentence lives in 4 places (index.html:3349, page prose :142, page FAQ :161, page JSON-LD :56); (4) three divergent inline engines (172/59/22 pages) prove independent generation events; (5) git history shows repeated whole-corpus patch commits (`0dced967`, `9248fb97`, `8a1f5ec7`) — each a bespoke migration for what should be `make build`.

Corollaries the user should internalize:
- The monolithic index.html is **not** the main problem. It is fragile but it is *one* file, has self-tests, and CLAUDE.md correctly fences it off. The maintenance catastrophe is in /pages/.
- "Shared components" at *runtime* (framework, JS includes) is the wrong layer for a static SEO site; sharing must happen at **authoring time**. Page self-containment is a feature worth keeping — the goal is self-contained *outputs* from shared *sources*.
- The second real problem is strategic, and the repo already knows it (CLAUDE.md "SEO reality"): page volume is running far ahead of domain authority. Engineering work that makes pages cheaper to mass-produce is only valuable if regeneration (freshness, correctness) — not proliferation — is the goal.

## Phase 15 – Solution Design

### 15.0 The CLAUDE.md tension, addressed head-on

CLAUDE.md forbids: changing calculator math/IDs/country logic, refactoring index.html into modules, and (implicitly, "no build step") introducing a framework. **The design below honors all three by moving the build to the *authoring* side:** a generator that runs on the developer's machine (exactly like `generate_pages.py` already does today) and commits plain static HTML. Netlify continues to deploy verbatim — no build command, no framework, no change to index.html initially. If the owner later wants Netlify to run the generator, that's a one-line `[build] command` — optional, not required.

### 15.1 Target architecture

```
sources/  (NEW — the single source of truth, all JSON/MD)
├── registry/countries.json      34 profiles: rates, bank arrays, product names/notes, defaults, flags
├── registry/products.json       15 products: formulas ref, embed type, FAQ templates, title/desc patterns
├── registry/pages.json          ~310 page specs: slug, product, country|bank|amount, cluster, category tags
├── registry/banks.json          340 rows (or folded into countries.json)
├── templates/                   layout.html, embed/{emi,car,sip,fd,edu,zakat,convert,...}.html, partials/{head,hreflang,related,faq,footer}
├── engine/embed.js              ONE engine (superset of the 3 variants), emitted as a shared file
└── content/                     per-page unique prose overrides (markdown), for pages that need hand-written sections

build.py (or build.mjs — extends the proven generate_pages.py pattern)
  → emits pages/*.html, pages/index.html (directory), sitemap.xml, hreflang meshes, related-links
  → runs formula self-tests + link checker + HTML diff report before writing
Deploy: unchanged (git push → Netlify serves static files)
```

- **Shared calculation engine:** one `sources/engine/embed.js` covering all 11 embed types; emitted either (a) inlined per page (preserves self-containment, still deduped at source) or (b) as `/pages/embed.js` referenced by all pages (gets the 1-year immutable cache; ~3 KB × N saved). Recommend (b) — the CSP on landing pages doesn't block same-origin scripts, and one HTTP request is a fair trade.
- **Shared country engine:** `countries.json` becomes the generator's input AND emits index.html's `COUNTRY_PROFILES/BANKS/SMART_DEFAULTS` block *when the owner is ready* — phase-gated because CLAUDE.md protects index.html; until then, a CI check merely *diffs* the JSON against index.html's objects to catch drift.
- **Shared metadata/SEO layer:** fix_seo.py's `make_title/make_description/make_related_links` logic ports directly into the templates (it already encodes the patterns — :341–636); FAQ JSON-LD and visible FAQ render from the same YAML/JSON fragment, killing duplication class #5.
- **Shared interlink engine:** related-links, hub country-blocks, hreflang meshes, guides-directory cards, and sitemap all render from `pages.json` — collapsing the four drift-prone registries (dup #8) into one. Adding country #35 becomes: 1 JSON entry + `python build.py` (regenerating the hreflang blocks in all sibling pages automatically — the O(n²) burden becomes O(1) human effort).
- **Shared validation:** build-time — schema-validate the registries, assert every internal href resolves to an emitted file (fix_seo.py's `add_link` existence check :503 generalized), run the EMI/SIP/FD reference tests from index.html's self-test expectations (:6874+) against the embed engine, W3C-validate a sample, fail the build on any error. Add this as a GitHub Action alongside indexnow.yml.
- **Dynamic routing / rendering strategy:** unchanged — pre-rendered static files. No SSR/ISR/hydration; the site's content is genuinely static between data refreshes. **State management:** unchanged in index.html (guardrail); the embed engine keeps its URL-param sync.
- **Performance moves (sequenced after generator):** move `#fc-embed-css` into landing.css (dup #3); shared embed.js; self-host fonts; *later, with explicit owner approval per CLAUDE.md*: extract index.html's data objects to a cached `/data.js` and raise HTML cache to 1 hour with asset fingerprinting.
- **Caching strategy fix:** keep HTML at 5 min; the win comes from moving weight out of HTML into immutable-cached assets, not from tuning the HTML TTL.

### 15.2 Alternatives compared

| | **A. Registry + in-repo generator (recommended)** | **B. Lightweight SSG (Eleventy/Astro)** | **C. Status quo (AI-session edits + patch scripts)** |
|---|---|---|---|
| Deploy model | unchanged: commit static HTML, Netlify serves | Netlify build step (or local build) — violates/needs amendment of CLAUDE.md "no build step" | unchanged |
| Fit to skillset | Python already proven in-repo (generate_pages.py, fix_seo.py); solo-maintainable | new toolchain (Node config, template language, data cascade) for one person who hasn't asked for it | none needed |
| Risk to SEO | zero if outputs are byte-diffed against current pages before first commit | medium: URL/markup churn risk during port | zero short-term; drift/corruption risk long-term (already materialized twice: commits 7be8127d, 8a1f5ec7) |
| Dedup power | full (all 14 duplication classes addressable) | full | none — every fix is another fix_seo.py |
| Incremental adoptability | high — start with 1 cluster (e.g. 33 car pages), keep the rest frozen | low — SSGs want to own the whole tree | n/a |
| Long-term ceiling | fine to ~5k pages; no ecosystem (i18n plugins etc.) | higher ceiling, ecosystem, real templating | editorial collapse ~500 pages |
| Verdict | **Choose A.** | Revisit B only if the site earns a custom domain + real traffic and a second maintainer | reject |

### 15.3 Migration roadmap (file-by-file, in order)

1. **Hygiene (1 commit, zero risk):** `git rm -r --cached node_modules` + gitignore it; remove `@anthropic-ai/sdk` from package.json; delete empty `css/`, `js/`; fix llms.txt "171→340 banks"; **remove the fabricated `aggregateRating`** from index.html:76–81 (SEO risk); reconcile the "no POST requests" FAQ line :2787 with GeoJS/lead-capture reality; fix zakat nisab fallback (:6092) to not assume INR.
2. **Extract the registries (read-only step):** script that *parses* index.html's `COUNTRY_PROFILES/BANKS/SMART_DEFAULTS` into `sources/registry/*.json`; hand-build `pages.json` from the filesystem + fix_seo.py's `classify()` (reuse its code — it already maps every slug). No emitted files change.
3. **Unify the embed engine:** write `sources/engine/embed.js` as the superset of the 3 variants; add reference tests mirroring index.html:6874 expectations.
4. **Template one cluster (pilot):** the 33 car-loan pages (they were the diff-proven mechanical cluster). Build `build.py` to emit them; **byte-diff against current files**; expected diffs only where variants disagree (engine unification) — review each class of diff once, then commit. Rollback = `git revert` (outputs are committed files; the old HTML is one commit away).
5. **Roll clusters in size order:** fd (34) → sip (34) → home (33) → zakat (16) → converters (21) → bank pages (56) → glossary (18) → hubs/amount/one-offs. Move `fc-embed-css` into landing.css during each cluster's port. After each cluster: run link checker + resubmit via existing IndexNow action (automatic on push).
6. **Generate the directory + sitemap from pages.json:** replace hand-maintained pages/index.html cards and fold generate_pages.py's sitemap function into build.py (keep DSCR generation there too). Retire fix_seo.py (its logic now lives in templates) — keep it in git history.
7. **CI hardening:** GitHub Action running `build.py --check` (registries valid, links resolve, emitted HTML matches committed HTML) on every PR/push.
8. **Only with explicit owner sign-off (CLAUDE.md gate):** emit index.html's data blocks from the registry (initially as a drift-checker, later as the writer); optionally extract them to a cached data.js. Never touch formulas/IDs — verified by keeping the in-page self-tests green and adding the same assertions to CI.

### 15.4 Risk analysis & rollback

- **Biggest risk:** silent markup drift during regeneration hurting rankings. Mitigation: byte-diff gate (step 4), cluster-at-a-time rollout, canonical/URL invariance (slugs never change), Search Console monitoring per cluster (**needs GSC access — unverifiable from repo**).
- **Rollback:** trivial by construction — every emitted page is a committed file; `git revert` restores the exact prior bytes; Netlify redeploys in minutes; no data migrations exist anywhere in this system.
- **Human risk:** the owner maintains via AI sessions; the registry + build script is *more* AI-session-friendly than 300 frozen files (small JSON edits, deterministic output), which is the strongest practical argument for Option A.
- **What NOT to do:** do not add more page volume until existing pages index (CLAUDE.md's own advice); do not refactor index.html's calculators; do not introduce a runtime framework for a site whose entire dynamic behavior is 2 formulas and a currency picker.
